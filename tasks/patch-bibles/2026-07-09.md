### WP-3 · #43 [P0] Recurring Keychain password prompt — Claude Code (Opus) + /security-review
- **Worktree:** `../tokei-worktrees/2026-07-09-keychain-no-prompt` (branch `patch/2026-07-09/keychain-no-prompt`)
- **Scope (in):**
  - `AIUsageDashboardApp/Core/Network/ClaudeUsageClient.swift` — `DefaultClaudeUsageCredentialsReader`
    (`readKeychainData()` / `readCredentials()`, cited at `:91-114`).
  - `AIUsageDashboardApp/Core/Security/Keychain.swift` — only if a shared spawn/read helper belongs here.
  - `Tests/ClaudeUsageTests/…` + `Tests/Fixtures/…`.
- **Scope (out):** UI, App, providers (WP-1/WP-2), all other Core. Do not touch `ProviderID.swift`.
- **Root cause (from #43, diagnosed):** Tokei calls `SecItemCopyMatching` from the app on the
  generic-password item **`"Claude Code-credentials"`** — owned/recreated by the Claude Code CLI on
  every OAuth refresh. Foreign-item ACL → macOS prompts every read; "Always Allow" is wiped each time
  the CLI recreates the item; the 3-query fallback multiplies the dialogs; cancel is never respected.
- **Fix — adopt TokenTracker's proven zero-dialog read (prior art researched 2026-07-09,
  `mm7894215/TokenTracker` `src/lib/subscriptions.js`):**
  1. **File first (cheap, cross-platform):** read `~/.claude/.credentials.json` if present, parse
     `claudeAiOauth.accessToken`. **Note (verified on this Mac): the file does NOT exist on macOS** —
     Claude Code stores creds only in Keychain here — so this step is a no-op on macOS but correct for
     Linux/Windows and future-proofing. It is NOT the load-bearing fix on macOS.
  2. **macOS load-bearing fix — spawn the `security` CLI, not `SecItemCopyMatching`:**
     ```
     /usr/bin/security find-generic-password -s "Claude Code-credentials" -w
     ```
     via `Process`, `stdio` = stdin ignore / stdout pipe / **stderr discarded**, ~2s timeout.
     Non-zero exit, spawn error, or timeout → return `nil` **quietly** (no throw, no UI block, no
     retry). Trim stdout; empty → nil. Parse `claudeAiOauth.accessToken` from the JSON payload
     (same shape already decoded). **Why this works:** the item's ACL already trusts the `/usr/bin/security`
     binary (the same context Claude Code used to create it), and that trust attaches to `security`,
     not to Tokei — so it survives the CLI deleting+recreating the item on every token rotation.
     TokenTracker exercises exactly this (`readMacosKeychainPassword`, `-w`) with zero dialogs in the wild.
  3. **Else** → calm disconnected state: no dialog storm, no retry until the user acts or relaunch.
  4. **Delete the 3× fallback query multiplier** (`keychainQueries`) — one `security` read, not three
     `SecItemCopyMatching` attempts. Respect a failure as final for the cycle.
- **Acceptance criteria:**
  - [ ] No `SecItemCopyMatching` against the foreign `"Claude Code-credentials"` item remains in the
        read path; replaced by file-first → `security -w` spawn.
  - [ ] The 3-query fallback storm is gone; a single failed read is final for the cycle (no re-prompt loop).
  - [ ] Cancel/failure → quiet nil + calm "Claude not connected — reconnect" surface (no dialog storm).
  - [ ] **Live proof (non-negotiable, from #43's checklist):** on this machine, running
        `security find-generic-password -s "Claude Code-credentials" -w` and the app's new read path
        returns the token with **no GUI prompt**. Metadata probes never prompt — the `-w` path is the
        one that must be exercised and shown dialog-free. Document the observed result in §8.
  - [ ] Tokei's own Keychain items (service `ai.AIUsageDashboard`, e.g. Cursor cookie) are untouched.
- **Test requirements:**
  - [ ] Inject the process runner (mirror TokenTracker's `securityRunner` seam) so tests can simulate
        exit 0 + payload, non-zero, and timeout without hitting the real Keychain.
  - [ ] Test: file present → used, no spawn. File absent + spawn exit 0 → token parsed. Spawn non-zero
        / timeout → nil, no throw, no retry. Malformed JSON → nil.
  - [ ] `xcodebuild … -scheme AIUsageDashboardCore … test` green.
- **Design / UX constraints:** if you touch the "not connected" surface copy, keep it in the UI layer
  via existing state — do NOT build new UI here (that's WP-2's scope); a Core-emitted warning/auth
  status is the seam.
- **Known risks:**
  - Spawning `security` under the app sandbox / hardened runtime: verify `Process` to `/usr/bin/security`
    is permitted with the app's current entitlements (the app is Dev-ID signed + hardened-runtime +
    notarized as of v0.2.1). If the sandbox blocks `Process`, document it in §8 and fall back to the
    file path + calm disconnected state — never reintroduce the prompt storm. **This must be verified in
    the packaged/hardened build, not just a debug run.**
  - Timeout must be real (2s) so a hung `security` never blocks the sync loop.
- **Rollback:** revert merge commit; branch retained.
